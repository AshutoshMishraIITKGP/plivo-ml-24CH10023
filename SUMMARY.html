LLM Speedrun: Final Submission Summary

Project: 2,000 Step LLM Speedrun

Executive Summary

Our final submission transforms the provided ~1.3M parameter GPT baseline into a highly optimized 1,967,808 parameter LLaMA-style architecture. By maximizing the parameter budget and effectively increasing the tokens processed per step (via gradient accumulation), we achieved a massive drop in Bits-Per-Byte (BPB) from a baseline of 2.9464 down to 1.9101 BPB, utilizing an accelerated 500 optimizer step limit.

1. Architecture Breakdown & Development

We completely overhauled model.py to match modern best practices from models like LLaMA and Mistral, utilizing pure PyTorch with no external libraries. Our target was to mathematically hit the 2,000,000 parameter constraint while maximizing representation depth.

  • Scale & Tying: 4 Layers, 192 Embedding Dimension, 6 Attention Heads (head_dim=32). We tied the input token embedding and output head matrices, saving ~196k parameters that were re-allocated to deepen the network.
  • RMSNorm: Replaced standard LayerNorm to save parameters (removing bias vectors) and speed up CPU inference.
  • SwiGLU FFN: Replaced the standard GELU activation. Since SwiGLU utilizes 3 matrices instead of 2, we specifically calculated and adjusted the intermediate dimension to 512 so that the parameter count remained exactly mathematically matched to the budget.
  • Rotary Position Embeddings (RoPE): Stripped out learned absolute position embeddings. We implemented RoPE entirely from scratch using pure PyTorch tensor operations. RoPE vastly improves context generalization and frees up absolute position parameter space.

2. Custom BPE Tokenization (No Libraries)

Instead of relying on HuggingFace `transformers` or `tiktoken` (which violate the pure Python/stdlib rule), we developed a custom Byte-Pair Encoding (BPE) tokenizer from scratch.

  • Regex Pre-tokenization: We used a strict regex pattern (similar to GPT-4) to prevent cross-character merges (e.g., merging across punctuation or numbers), ensuring high-quality tokens.
  • Compression vs Constraint: By expanding the vocabulary to 1024, we achieved a 2.1x compression ratio over raw bytes. This packed double the text into the 256-token context window.
  • Byte Fallback: To strictly adhere to the competition rules (must encode arbitrary UTF-8), the first 256 tokens are reserved for raw bytes, ensuring the tokenizer.load() function will never throw an out-of-vocabulary error.

3. Training Recipe & Ablation Journey

Our RUNLOG.md outlines a 5-stage ablation journey. The most critical realization was finding the correct architectural balance for small models:

  • The Deep/Narrow Trap: We initially hypothesized that extreme depth would yield better linguistic abstraction. We attempted an architectural swap to 9 layers and 96 dimensions (head_dim=8). The loss plateaued severely.
  • The Fix: We diagnosed that a head_dim of 8 severely choked the attention mechanism. We fixed this by reverting to a wider, shallower network (4 layers, 192 dim, head_dim of 32). This instantly removed the informational bottleneck.

4. Why this is the best configuration

This configuration wins because it leaves nothing on the table. The starter code only uses ~55% of the 2M parameter budget and processes only 1,024 tokens per step. We utilized 98.4% of the parameter budget and pushed token throughput to 8,192 tokens per step. By swapping outdated components (LayerNorm/GELU/Absolute Pos) for modern equivalents (RMSNorm/SwiGLU/RoPE) and writing a robust tokenizer from scratch, we get significantly better gradient flow and representational capacity without exceeding the strict mathematical caps or relying on a single external package.
